# ソフト404対策 運用設計書（410運用）

最終更新日: 2026-03-06
対象: `oyakonojikanlabo.jp`（SPA + `.htaccess` 運用）

## 1. 目的

- ソフト404（実体のないURLが `200` になる状態）を段階的に解消します。
- 旧URLのうち移行先がないものは `410 Gone` を返し、検索エンジンの再クロール効率を改善します。
- 運用を属人化させないため、入力仕様・除外条件・生成手順・検証手順を固定化します。

## 2. 運用方針

- `301` を優先し、移行先がないURLのみ `410` にします。
- `.htaccess` は最終成果物を1ファイルに保ち、管理はCSVと生成物で分離します。
- いきなり全量投入せず、バッチ適用で誤爆リスクを下げます。

## 3. 入力CSV仕様

### 3.1 必須ファイル

- `unpublished.csv`（WordPress 非公開URL一覧）
- `gsc_soft404.csv`（Search Console soft404）
- `gsc_404.csv`（Search Console 404）
- `redirect_map.csv`（301用: `from,to`）
- `valid_urls.csv`（現行有効URL一覧。sitemap等から作成）

### 3.2 必須列

- 共通: `url`
- `redirect_map.csv` のみ: `from,to`

### 3.3 文字コード・形式

- 文字コード: `UTF-8`（BOM有無どちらでも可）
- 改行: `LF` または `CRLF`
- URL形式: パスのみ（例: `/sample-path/`）

### 3.4 正規化ルール

- スキーム/ドメインを除去してパス化
- 先頭 `/` を必須化
- 末尾 `/` を付与（ファイル拡張子付きURLは除く）
- 文字は小文字化
- クエリ・フラグメント（`?` `#`）を除去
- 重複URLを削除

### 3.5 受け入れ条件

- `url` が空行でない
- `http://` / `https://` の混入がない
- 相対記法（`../` など）がない

## 4. 除外ルール

### 4.1 `410` から除外（必須）

- `valid_urls.csv` に存在するURL（現行で `200` を返すURL）
- `redirect_map.csv` に含まれるURL（`301` 対象）
- 静的アセット配下
  - `/assets/`, `/icons/`, `/fonts/`, `/images/`, `/favicon`, `/manifest`
- システム系パス
  - `/wp-admin/`, `/wp-login.php`, `/wp-json/`（運用要件次第）

### 4.2 要レビュー除外

- 不正文字・不可視文字を含むURL
- `%` エンコードが壊れているURL
- 本来1件に統合すべき表記ゆれURL

### 4.3 固定ルール

- 既存の構造ルール（例: `category`, `author`, `page/[0-9]+` など）は継続利用
- 既存の個別 `301` は `410` より上位に配置

## 5. 生成手順

### 5.1 入力統合

1. `unpublished.csv` + `gsc_soft404.csv` + `gsc_404.csv` を結合
2. 正規化ルールを適用
3. 一意化

### 5.2 判定

1. `redirect_map.csv` に一致するURLを `301` 候補に分類
2. `valid_urls.csv` に一致するURLを除外
3. 残りを `410` 候補に分類
4. 要レビュー除外に一致するものを別ファイルへ退避

### 5.3 出力

- `out_301.conf`（個別301）
- `out_410.conf`（個別410）
- `out_410_skipped.txt`（要レビューURL）
- `generation_summary.txt`（件数サマリ）

### 5.4 `.htaccess` 反映

- 反映位置は SPA fallback より前
- 順序は以下で固定

1. HTTPS強制
2. 個別301（既存固定）
3. 構造301/410（既存固定）
4. CSV由来301
5. CSV由来410
6. 静的ファイル除外
7. SPA fallback

### 5.5 バッチ適用

- 初回は 50〜300件単位で分割反映
- 1バッチごとに検証してから次バッチへ進む
- 問題が出たバッチのみロールバック

## 6. 検証手順

### 6.1 事前検証（ローカル）

- `.htaccess` の構文チェック（可能なら）
- 生成件数が想定範囲か確認
  - `301` 件数
  - `410` 件数
  - 除外件数

### 6.2 反映直後検証（必須）

- `410` 候補から最低30件を抽出して `curl -I` で確認
  - 期待値: `410`
- 現行有効URLを最低30件確認
  - 期待値: `200`
- `301` 対象URLを最低10件確認
  - 期待値: `301`（移行先が正しい）

確認例:

```bash
curl -s -o /dev/null -w "%{http_code} %{url_effective}\n" -I "https://oyakonojikanlabo.jp/old-path/"
```

### 6.3 監視（反映後 1〜2 週間）

- Search Console の `soft 404` 件数推移
- 新規 `404` / `soft 404` の増減
- 主要ページのインデックス数異常有無

## 7. ロールバック手順

1. 問題のあるバッチ（またはブロック）を `.htaccess` から削除
2. 直前の安定版 `.htaccess` を再配置
3. `curl -I` で復旧確認
4. 原因URLを `out_410_skipped.txt` に移管し再発防止

## 8. 運用サイクル

- 月次運用（推奨）

1. 最新CSV取り込み
2. 生成
3. バッチ反映
4. 検証
5. Search Console の結果を次回除外ルールに反映

## 9. 完了条件（Definition of Done）

- 追加した `410` 対象URLのサンプル検証が全件 `410`
- 有効URLサンプルが全件 `200`
- 誤爆（本来 `200` のURLが `410`）が `0`
- 反映後、Search Console の soft404 が継続的に減少
